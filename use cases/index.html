# Створення нового облікового запису

ID: new_acc_reg
НАЗВА: Створення нового облікового запису
УЧАСНИКИ: Користувач, Система
ПЕРЕДУМОВИ: 1. Користувач не має облікового запису
РЕЗУЛЬТАТ: Новий обліковий запис
ВИКЛЮЧНІ СИТУАЦІЇ: 1. Обліковий запис даного користувача існує(new_acc_reg.acc_exist)
2. Пусті поля вводу данних (new_acc_reg.emp_fields)
ОСНОВНИЙ СЦЕНАРІЙ: 1. Користувач починає взаємодіяти з продуктом.
2. Користувач натискає на кнопку "Почати реєстрацію"
3. Користувач вводить реєстраційні дані
4. Користувач натискає на кнопку "Створити обліковий запис"
5. Система перевіряє існування облікового запису даного користувача.
6. Система створює новий обліковий записю
7. Користувач закінчує взаємодію.

# Авторизація в системі

ID: acc_auth
НАЗВА: Авторизація користувача
УЧАСНИКИ: Користувач, Система
ПЕРЕДУМОВИ: 1. Користувач має обліковий запису
РЕЗУЛЬТАТ: АВторизація в системі
ВИКЛЮЧНІ СИТУАЦІЇ: 1. Користувач не має облікового запису(acc_auth.acc_not_exist)
2. Не правильно введені дані (acc_auth.wrong_fields)
ОСНОВНИЙ СЦЕНАРІЙ: 1. Користувач натискає кнопку "Увійти".
2. Користувач вводить дані для входу.
3. Користувач натискає кнопку "Увіти"
4. Система превіряє наявність облікового запису та превильність введених даних
5. Система авторизує користувача.
6. Користувач працює в системі від імені свого обліковго запису.

# Запит на пошук даних

ID: search_data
НАЗВА: Пошук набору даних
УЧАСНИКИ: Користувач, Система
ПЕРЕДУМОВИ: авторизувався в системі
РЕЗУЛЬТАТ: Користувач отримує данні за якими надав запит пошуку.
ВИКЛЮЧНІ СИТУАЦІЇ: 1. Даних за запитом не існує.(search_data.data_not_exist)
ОСНОВНИЙ СЦЕНАРІЙ: 1. Користувач вводить заипт на бажані дані в пошукове поле.
2. Користувач натискає конпку "Знайти"
3. Система перевіряє наявність даних за запитомю
4. Система превіряє показує що вона знайшла за запитом користувача.

# Візуалізація даних

ID: vis_data
НАЗВА: Візуалізація даних
УЧАСНИКИ: Користувач, Система
ПЕРЕДУМОВИ: Користувач увійшов в систему та обрав дані.
РЕЗУЛЬТАТ: Користувач отримує таблицю або графік на основі обраних даних.
ВИКЛЮЧНІ СИТУАЦІЇ: Дані не можна візуалізувати(подати у виді графіку або таблиці)(vis_data.data_unvis)
ОСНОВНИЙ СЦЕНАРІЙ: 1. Користувач обирає дані статистику по яких хоче отримати.
2. Користувач натискає кнопку "Побудувати графік/таблицю", якщо така функція є.
3. Система будує графік або таблицю
4. Користувач завершує взаємодію.

# Запит на отримання прав редагування контенту

ID: zap_edit
НАЗВА: Запит на отримання прав редактора
УЧАСНИКИ: Користувач, Адміністратор, Система
ПЕРЕДУМОВИ: Користувач увійшов в систему
РЕЗУЛЬТАТ: Користувач отримує права на редагування
ВИКЛЮЧНІ СИТУАЦІЇ: Користувач не підходить на роль редактора(zap_edit.usr_unsuit)
ОСНОВНИЙ СЦЕНАРІЙ: 1. Користувач надсилає запит на отримання прав редактора
2.Адміністратор розглядає запит та приймає його або відхиляє
3. Відповідно до реакції адміністратора на запит система оновлює статус користувача що надсилав його

# Завнтаження нових даних

ID: data_upload
НАЗВА: Завантаження набору нових даних
УЧАСНИКИ: Адміністратор, Система
ПЕРЕДУМОВИ: Адміністратор авторизований у системі
РЕЗУЛЬТАТ: Завантажений набір даних у базі даних
ВИКЛЮЧНІ СИТУАЦІЇ: Завантажувані дані вже є в системі(data_upload.data_exist)
ОСНОВНИЙ СЦЕНАРІЙ: 1. Адміністратор натискає на кнопку "Завантажити нові дані"
2. Адміністратор обирає дані, що хоче завантажити"
3. Адміністратор натискає кнопку "Завантажити обрані дані"
4. Система перевіряє наявність таких даних
5. Система завантажує нові дані у базу даних

# Видалення даних

ID: data_del
НАЗВА: Видалення набору даних
УЧАСНИКИ: Адміністратор, Система
ПЕРЕДУМОВИ: Адміністратор авторизований у системі та знайшов неактуальні чи не правдиві
РЕЗУЛЬТАТ: Набір даних видалений з бази даних
ВИКЛЮЧНІ СИТУАЦІЇ: Дані вже видалені із бази(data_del.data_not_exist)
ОСНОВНИЙ СЦЕНАРІЙ: 1. Адміністратор натискає на кнопку "Видалити дані"
2. Адміністратор обирає дані, що хоче видалити"
3. Адміністратор натискає кнопку "Видалити обрані дані"
4. Система видаляє дані з бази даних

# Надання прав редактора коритувачу

ID: editor_add
НАЗВА: Надати права редактора користувачу
УЧАСНИКИ: Адміністратор, Система
ПЕРЕДУМОВИ: Адміністратор авторизований у системі
РЕЗУЛЬТАТ: Користувач отримує права редактора
ВИКЛЮЧНІ СИТУАЦІЇ: У користувача вже були права редактора(editor_add.editor_exists)
ОСНОВНИЙ СЦЕНАРІЙ: 1. Адміністратор надає запит на користувача
2. Система показує користувача на якого був запит
3. Адміністратор натискає на кнопку "Надати права редактора" навпроти даних користувача"
4. Користувач отримує права редактора
5. Адміністратор завершує взаємодію з системою

# Зняття прав реадктора з користувача

ID: editor_del
НАЗВА: Анулювати права редактора для користувача
УЧАСНИКИ: Адміністратор, Система
ПЕРЕДУМОВИ: Адміністратор авторизований у системі
РЕЗУЛЬТАТ: Користувач втрачає права редактора
ВИКЛЮЧНІ СИТУАЦІЇ: У користувача немає прав редактора(editor_del.usr_not_editor)
ОСНОВНИЙ СЦЕНАРІЙ: 1. Адміністратор надає запит пошуку всіх редакторів
2. Система у вигляді списку показує всіх редаторів
3. Адмінстратор обирає потрібного редактора та натискає кнопку "Забрати права редактора"
4. Користувач втрачає права редактора
5. Адміністратор завершує взаємодію з системою

# Редагування набору даних

ID: data_edit
НАЗВА: Редагувати набір даних
УЧАСНИКИ: Редактор, Система
ПЕРЕДУМОВИ: Редактор авторизований у системі, обраний набір даних є у базі даних
РЕЗУЛЬТАТ: Редагований набір даних
ВИКЛЮЧНІ СИТУАЦІЇ: Зміни скасовано(data_edit.edit_error)
ОСНОВНИЙ СЦЕНАРІЙ: 1. Редактор надає запит на дані, що хоче редагувати
2. Система показує список даних за запитом редактора
3. Редактор натискає на кнопку "Редагувати" навпроти даних
4. Відкривається форма редагування
5. Редактор редагує дані
6. Редактор зберіє дані натиснувши кнопку "Зберігти зміни"
7. Система вносить зміну у базу даних
8. Редактор закінчує взаємодію
